# 网页多模态 Agent 的设计与实现：系统设计与实验报告

---

## 摘  要

随着大语言模型（Large Language Model, LLM）和视觉语言模型（Vision-Language Model, VLM）的快速发展，基于自然语言指令自主操控浏览器完成复杂网页任务的多模态智能代理（Agent）逐渐成为人机交互与自动化研究的前沿方向。与传统的脚本自动化相比，网页智能代理需要在开放、动态、弱结构化的环境中持续决策：一方面要理解页面的视觉布局、控件语义与状态变化，另一方面要将高层目标拆解为可执行的点击、输入、滚动等细粒度动作序列。这类任务同时涉及多模态感知、长程规划、工具调用与错误恢复，具有典型的综合性与挑战性。

本文面向真实网页环境中的多步任务自动化问题，设计并实现了一个基于 LLM-VLM 双层协作架构的网页多模态 Agent 系统——ClassicWebAgent。系统以 CoALA 认知架构为总体组织框架，将智能体能力划分为记忆层、动作空间层和决策循环层三个层次。在感知层面，系统采用基于 Chrome DevTools Protocol (CDP) 的三流融合方案，将 DOM 骨架、Accessibility 语义树与布局快照对齐到统一的增强 DOM 树，并以 `backendNodeId` 贯穿采集、序列化与执行全链路。在决策层面，系统采用 ReAct 闭环范式，由 VLM SubAgent 在子任务内部自治执行"观察—规划—执行—验证"的迭代决策流程，由 LLM Director 负责任务分解、子任务调度与最终报告生成。系统定义了覆盖元素交互、页面操作、导航、标签页管理和信息获取五大类别的 16 个外部动作以及 5 个内部推理和记忆动作，并实现了三层记忆机制以支持长链任务中的上下文保持。

实验阶段设计了覆盖天气、出行、购物、体育、金融、资讯、学术 7 个领域的 10 个端到端测试用例，建立了融合任务完成度、信息质量和子任务完成率的三维度加权评估体系。初步实验结果表明，系统在任务完成度方面表现良好，平均得分 90.0 分（满分 100 分），但在 VLM 子任务执行层面仍面临复杂表单交互场景下的可靠性挑战，子任务成功率仅为 21.4%。本文对实验结果进行了详细的失败模式分析，并提出了针对性的改进方向。

**关键词**：多模态智能代理；大语言模型；视觉语言模型；CoALA 架构；ReAct 范式；浏览器自动化；Chrome DevTools Protocol

## 1 绪论

### 1.1 选题背景与意义

随着大语言模型（LLM）和视觉语言模型（VLM）的快速发展，基于自然语言指令完成网页操作的智能体逐渐成为人机交互与自动化研究的重要方向[1-4]。与传统的脚本自动化相比，网页智能体需要在开放、动态、弱结构化的环境中持续决策：一方面要理解页面的视觉布局、控件语义与状态变化，另一方面还要将高层目标拆解为可执行的点击、输入、滚动等动作序列。这类任务同时涉及多模态感知、长程规划、工具调用和错误恢复，具有典型的综合性与挑战性。

一系列研究实践表明，网页多模态智能体的设计，既要关注智能体认知结构的构建，也要考虑智能体与网页环境的交互机制，还要考虑模型能力与实际应用需求的匹配，是一个综合性很强的研究问题[5-8]。如何设计出高效、泛用性强、鲁棒性好的网页智能体，既具有重要的理论研究意义，也具有广泛的应用前景。

本课题的意义体现在三个层面。第一，研究意义：围绕网页多模态智能体的核心能力——动作规划、视觉理解、网页交互——设计智能体的系统架构与模块实现，探索其在真实网页环境中的表现与局限，能够帮助我们深入了解智能体设计范式与能力边界，掌握多模态智能体设计和优化的相关经验。第二，应用意义：面向信息检索、表单填写、流程办理等高频网页任务，让智能体代替用户完成操作能够显著提升效率和用户体验；而使智能体获得网页界面理解和操作的能力，也能进一步拓宽智能体的能力边界，为智能体执行更多复杂任务奠定基础。第三，工程意义：设计并实现一个可用的网页多模态智能体系统，特别是其认知架构、交互机制和错误恢复能力的实现，能够为智能体系统的工程实践提供重要的经验和技术积累，具有较强的工程启发意义。

### 1.2 国内外研究现状和相关工作

#### 1.2.1 从 CoALA 与 ReAct 出发的智能体结构研究

近年来，智能体研究逐步从"单次问答模型"转向"具备持续决策能力的认知系统"。Sumers 等人[5]提出的 CoALA（Cognitive Architectures for Language Agents）框架将语言智能体划分为记忆层（Memory）、动作空间层（Action Space）与决策循环层（Decision Cycle）三类核心要素，为智能体的设计提供了明确的认知结构指导，为包括网页智能体在内的多模态智能体结构构建了统一的描述体系。

在工程实践层面，Yao 等人[2]提出的 ReAct 范式通过"Thought-Action-Observation"交替推进，将推理与行动耦合到同一闭环中，显著提升了多步任务中的可解释性与纠错能力。相比仅一次性生成完整步骤，ReAct 更适合动态网页环境：模型可根据每一步观察结果调整下一步动作，降低早期错误在长链路任务中的放大效应。绝大多数网页智能体都采用了 ReAct 或其变体作为核心设计范式，表明了该范式在实际应用中的有效性。

#### 1.2.2 WebAgent 架构与范式演进

WebAgent 近年来的研究呈现出从重量级操作接口向轻量化、拟人化设计演进的趋势。

**Mind2Web**[6] 首次在真实网页环境下系统构建了"通用网页智能体"的数据基础（覆盖多网站、多领域、多步任务），并提出 MindAct 两阶段范式：先由小模型筛选候选元素，再由大模型在候选集合中执行动作决策。该工作将 WebAgent 研究从简化模拟环境推进到真实网页场景，并明确了"元素检索 + 动作决策"的基础问题分解，为后续多模态网页智能体方法提供了统一起点。

**WebVoyager**[3] 将网页截图与可交互元素文本联合输入模型，并采用 Set-of-Marks（SoM）对元素进行编号标注，使模型可以在视觉上下文中稳定引用目标元素。这种设计弥补了单纯文本或视觉输入的不足，提升了模型对复杂页面结构的理解与定位能力。

**SeeAct**[4] 强调能够正确定位目标元素是网页智能体成功的前提，提出了视觉-结构融合的定位方法，而非单独依赖任一模态。该工作说明：在真实网页中，DOM 提供可执行结构约束，截图提供布局与语义线索，二者互补是提升定位准确率与任务成功率的必要条件。对于不具有微调能力的模型，视觉-结构融合的输入设计是提升性能的关键。

**BrowserAgent**[7] 和 **WebSight**[8] 这两个较新的研究强调视觉优先的网页理解与交互范式，不依赖 SoM 标注和 DOM 元素信息，更强调拟人化的环境感知和操作行为模拟。这些工作还减少了对复杂浏览器操作指令的依赖，只提供简洁的必要动作集，这也更贴近人类的操作习惯。但值得注意的是，这些研究的成果与其进行的特化模型训练和微调密切相关，视觉优先的设计在未经微调的通用模型上的有效性仍有待验证。

**See and Remember（V-GEMS）**[9] 面对 LLM 在长上下文处理中的挑战，提出引入状态栈与符号化工具，重点解决长程导航中的"迷失、循环与计数误差"问题。其创新点在于将操作路径与历史状态显式建模，通过 URL 栈等手段增强模型的环境记忆与状态跟踪能力，显著提升了长链路任务中的成功率和稳定性。

在开源工程实践方面，**browser-harness**[10] 提供了一个面向真实浏览器交互的轻量工程范式。其核心思想是以最小中间层直接连接浏览器操作接口与智能体，减少不必要的抽象，使智能体能够更直接地感知和操作网页环境，并引入自愈机制和 Skill 库机制。这些特质与当前智能体的设计潮流高度契合，具有较强的实用价值和工程启发意义。

#### 1.2.3 WebAgent 评测标准设计

评测方面，**VisualWebArena**[2] 提供了重要的方法学创新。该基准在真实网页环境中构建大规模视觉锚定任务，并将评测从单一"是否完成"扩展到更细粒度的函数化判定（如文本匹配、语义匹配、视觉问答与图像匹配等），显著提升了开放任务评估的可操作性与可解释性。其贡献在于推动网页 Agent 评测从"结果导向"走向"过程可分析"，这也为本课题后续实验中的指标设计与错误分类提供了重要参考。

综合而言，WebAgent 近年来的设计范式逐步由重量级的操作接口定义和完备的标注系统，向着更轻量、拟人化的设计转变。同时，对特化模型训练的依赖正在逐渐降低，未来更通用、适应性更强的智能体设计范式将成为研究的重点。

## 2 系统设计与实现

本章围绕网页多模态 Agent 的系统设计方案展开，重点说明系统目标、总体架构、关键模块的设计细节及工程实现路径。

### 2.1 系统目标与设计原则

本项目面向真实网页环境中的多步任务执行，目标是在自然语言指令驱动下，通过大模型路径规划、多模态融合环境感知、闭环决策与实时验证，实现高效、精确的网页交互，继而完成用户指定的复杂任务。

系统设计遵循以下七项原则：

**原则一：多模态融合。** 联合使用页面截图与 DOM 结构信息，通过多模态模型协作提高环境感知准确性，避免单模态信息缺失导致的错误。之所以不采用纯视觉方案，是因为没有经过微调的视觉模型对于网页结构的空间位置并不敏感[4]，往往需要结合 DOM 信息才能准确定位目标元素。本系统采用截图输入 VLM 进行视觉理解，同时将增强 DOM 树文本提供给模型作为结构化参考，实现视觉与结构的互补融合。

**原则二：多模型协作。** 系统使用两种大模型以适应不同执行过程的需要：VLM 负责环境感知与页面理解（战术层），LLM 负责任务分解与全局规划（战略层），利用模型间的互补优势提升系统鲁棒性与决策质量。在具体实现中，LLM 使用 deepseek-v4-flash 文本模型，VLM 使用 mimo-v2.5 视觉语言模型，二者通过 OpenAI 兼容 API 统一调用。

**原则三：闭环决策。** 采用"观察—规划—执行—验证"循环，利用 ReAct 范式[2]提升决策的可解释性与纠错能力，并支持在线修正。在子任务执行层面，VLM SubAgent 每步都经过完整的感知、规划、执行、验证四阶段，步间以 confidence 阈值机制筛选高置信度动作，低置信度时触发重规划。

**原则四：记忆机制。** 引入会话级记忆模块，记录历史动作与环境状态，支持长链任务中的上下文保持与错误恢复，避免重复探索。系统实现了三层记忆架构：observations 层记录子任务内的自然语言摘要，working 层记录最近操作步骤供 VLM 回溯，knowledge 层预留为跨任务的持久经验积累接口。

**原则五：轻量化设计。** 避免使用 LangChain 等重型框架，直接基于 Python 与 Playwright 实现核心功能，保持系统的简洁性与可控性，便于调试，防止中间层过多导致的性能损失与错误传播。系统整体代码量控制在合理范围内，所有模块优先单文件实现，目录深度不超过三层。

**原则六：可诊断性。** 通过规范化的日志结构输出结构化轨迹与错误分类，支持失败归因与实验分析。每次运行在 `log/` 目录下创建独立子目录，内含完整运行日志（`run.log`）、最终报告（`report.md`）、子任务状态文件（`sub_tasks.json`），以及当开启 trace 模式时保存的每步截图和 DOM 树快照。

**原则七：易拓展性。** 系统设计考虑未来的功能拓展，在架构层面预留了 Skill 库模块和长期记忆模块的接口，并通过 `extra: dict` 字段承载动作扩展参数，新增动作类型不改变 `Action` 数据结构的定义。

### 2.2 总体架构

#### 2.2.1 架构组织：CoALA 框架

本系统采用 CoALA 认知架构[5]作为总体架构组织方式，将智能体能力划分为三个层次：

1. **记忆层（Memory）**：包含 observations（观察记忆）、working（工作记忆）和 knowledge（知识记忆，预留）。工作记忆保存当前任务目标、页面状态与最近动作；观察记忆保存 VLM 子任务执行过程中的自然语言摘要。

2. **动作空间层（Action Space）**：定义外部动作与内部动作。外部动作对应浏览器操作（点击、输入、滚动等，共 16 个），内部动作对应推理、检索与记忆更新（共 5 个），合计 21 个动作类型。

3. **决策层（Decision Cycle）**：基于当前观察与记忆状态，由 LLM Director 进行任务分解和子任务调度，由 VLM SubAgent 在子任务内部进行 ReAct 闭环执行，持续循环直至完成任务。

#### 2.2.2 双层协作机制

系统采用创新的 LLM-VLM 双层协作架构，将战略级任务规划与战术级浏览器操控分离到不同的模型层。其核心工作流程如下：

```
用户任务（自然语言字符串）
  │
  ▼
┌─────────────────────────────────────────┐
│          LLM 主代理层 (Agent)            │
│                                          │
│  Phase 1: Director.plan(task)            │
│    输入：用户任务                         │
│    输出：task_plan + todo_list            │
│          + first_sub_task                │
│                                          │
│  Phase 2: 执行调度循环                    │
│    for each sub_task:                    │
│      SubAgent.run(sub_task)              │
│        → observations                   │
│      Director.review(observations)       │
│        → update todo_list                │
│        → next_sub_task / done            │
│                                          │
│  Phase 3: Director.report()              │
│    输入：task_plan + all_observations     │
│    输出：最终报告 (Markdown / HTML)        │
└──────────────┬──────────────────────────┘
               │ SubAgent.run()
               ▼
┌─────────────────────────────────────────┐
│        VLM 子代理层 (SubAgent)           │
│                                          │
│  ReAct 自治循环 (最多 50 步):             │
│    Perception.observe() → PageState      │
│    Planner.plan() → Action[]             │
│      含 confidence 阈值判定               │
│    Executor.execute() → ActionResult     │
│    Memory → observations 累积            │
│    直到 DONE / FAIL / 超步数              │
└──────────────┬──────────────────────────┘
               │ Playwright API
               ▼
         [Chromium 浏览器]
```

**LLM 层（Director）**负责战略决策。当收到用户任务后，Director 首先调用 LLM 将一句话需求拓展为详实的任务计划书（task_plan），内容包括研究维度定义、目标数据源（含具体 URL）、子领域或分类划分、时间范围、数量目标和评判标准等。同时，LLM 从计划书中提炼出可执行的子任务清单（todo_list），并以规范的 JSON 格式输出。每个子任务被编写为一段清晰的自然语言描述，派发给 VLM SubAgent 执行。

Director 通过持久对话上下文维护任务状态。SubAgent 返回的 observations 以 UserMessage 形式追加到 Director 的消息历史中，因此 LLM 能持续"感知"所有历史信息，动态调整后续计划。这种设计使得系统支持三种子任务调度模式：串行依赖模式（后续子任务依赖前置结果）、并行独立模式（多源搜索等无依赖场景，架构已预留接口）和混合模式。

**VLM 层（SubAgent）**负责战术执行。SubAgent 对外暴露唯一接口 `run(sub_task) → observations`，类似于异步函数调用——接收子任务描述，返回执行结果文本。内部自动创建 Perception、Planner、Executor 三个组件，构成完整的 ReAct 闭环。SubAgent 运行在单标签页的浏览器会话中，在子任务执行期间不清除当前页面状态，支持跨步的状态连续性。

每步 VLM 输出包含六个字段：`thinking`（推理过程）、`verification`（对当前状态的确认）、`memory`（关键发现摘要，自动追加到 observations）、`goal`（当前步骤目标）、`actions`（动作序列）和 `confidence`（置信度，0-1 之间的浮点数）。confidence 阈值机制是执行可靠性的核心保障：当 confidence 大于等于阈值（默认 0.9）时直接执行；低于阈值时触发重规划（不执行动作）；连续三次均不达标则返回 FAIL。

### 2.3 关键模块设计与实现

#### 2.3.1 感知模块

感知模块负责聚合截图语义与 DOM 结构信息，在 ReAct 闭环中承担 Observe 环节的核心职责。其技术路线为：CDP 做数据采集与元素解析，Playwright 做动作执行，`backendNodeId` 作为贯穿采集、序列化与执行全链路的唯一标识符。

**CDP 三流采集。** 感知模块通过三个 CDP 协议方法并行获取页面信息。`DOM.getDocument({depth: -1})` 获取完整的节点层级结构、父子关系和 HTML 属性；`Accessibility.getFullAXTree()` 获取无障碍语义信息（role、name、properties 如 disabled/checked/expanded 等）；`DOMSnapshot.captureSnapshot()` 获取布局数据（bounds 坐标、computed styles、is_clickable 可点击标记）。三条数据流通过 `backendNodeId` 对齐到同一棵增强 DOM 树。

**增强 DOM 树构建。** 以 CDP 返回的 DOM 树为骨架，在递归构建过程中为每个元素节点挂载 AX 语义数据和 Snapshot 布局数据。构建过程同步完成可见性判定和可交互检测：满足以下任一条件的节点标记为 hidden，其整个子树被跳过——HTML `hidden` 属性、`aria-hidden="true"`、CSS `display:none` 或 `visibility:hidden` 等内联样式、AX `hidden` property 为 true、`input[type=hidden]`、无用标签（`style`/`script`/`meta`/`noscript`）、CSS class 名含 `hide`/`hidden` 等。

**五级可交互检测。** 未隐藏的元素节点按以下五级条件检测交互性，满足任一即标记为可交互：（1）原生交互标签（`a`/`button`/`input`/`select`/`textarea`/`label`/`details`/`summary`/`dialog`）；（2）AX 交互型 role（`button`/`link`/`checkbox`/`radio`/`textbox`/`combobox`等 17 种）；（3）DOM 用户交互事件属性（`onclick`/`onmousedown`/`onkeydown`等 17 种）和可聚焦属性（`tabindex`）；（4）AX 交互属性（`focusable`/`editable`/`checked`/`expanded`/`pressed`/`selected`）；（5）CSS `cursor:pointer`（通过 AX 属性间接判断）。此外，AX `disabled` property 为 true 的元素保留在交互列表中但标记为不可交互。

**序列化输出。** 增强 DOM 树按深度优先递归序列化为文本格式，遵循以下规则：可交互元素输出 `[backendNodeId]<tag attr/> @(x,y,w,h)` 格式；不可交互容器不输出标签行但保留子节点层级缩进；容器处理完所有子节点后插入空行作为语义分组边界；隐藏节点子树直接跳过。该序列化文本将与页面截图（经 PIL optimize PNG 压缩为 base64 data URI）一起作为 VLM 的双模态输入。

**元素定位闭环。** `backendNodeId` 作为 CDP 协议的原生标识符贯穿全程。执行阶段，系统通过 `DOM.resolveNode({backendNodeId})` 解析节点，随后在节点上执行 JavaScript 生成唯一 CSS 选择器路径（优先使用 id，其次组合 tag + class + nth-of-type），最终通过 Playwright `locator()` API 获取 ElementHandle 执行原子操作。这一设计避免了自行维护额外的编号映射表。

#### 2.3.2 规划与执行模块

**VLM 动作规划（Planner）。** 规划器接收当前 PageState（截图 + tree_text + URL + 标题）和子任务描述，构建包含 system prompt（从 `planner.yaml` 加载）、文本内容（模板填充）和截图（作为 `image_url` content part）的请求发送给 VLM。VLM 需输出严格的 JSON 格式响应，包含 thinking、verification、memory、goal、actions 和 confidence 六个字段。规划器对返回的 JSON 进行完整性校验——必须包含所有六个必填字段、actions 数组中的每个动作的 type 必须在规定的 21 个动作名称集合内——校验失败时放弃本次结果并触发重试。

**动作执行（Executor）。** 执行器采用典型的策略模式，将 21 个动作类型路由到 21 个独立的 `_execute_*` 方法。外部动作通过 `backendNodeId` 经 CDP 定位到 ElementHandle 后执行 Playwright 原子操作；内部动作直接操作 Memory 或返回终端信号。所有异常在 Executor 层被捕获并转换为 `ActionResult(success=False)`，不会向上抛出。

执行器中包含若干工程化的容错设计：WAIT 动作的超时（10 秒，低于 Playwright 默认 30 秒）不等同于失败，返回 success 让 VLM 根据下一步截图自行判断页面状态；GOTO 动作在连续 3 次失败后自动跳过当前 URL，防止无限重试不可达地址；CLICK 动作执行后自动检测标签页数量变化，若检测到 popup 新标签页，则终止当前动作序列的剩余动作，在下一次 ReAct 循环中让 VLM 基于新标签页的 DOM 树重新决策。

**子任务自治循环（SubAgent Core）。** SubAgent 是三个内部组件的编排器，对外暴露 `run(sub_task) → observations` 单一接口。每次执行前清空上一子任务的 observations 和 working 记忆。主循环在每次迭代中依次调用 Perception.observe()、Planner.plan()、Executor.execute()，并将执行结果文本作为 VLM 下一步决策的上下文反馈。循环在以下条件下终止：VLM 输出 DONE 或 FAIL 动作、达到最大步数限制（50 步）、VLM 返回空动作序列。子任务的最终 completion status 被记录在 `current_status` 字段中，供 Agent 保存到 `sub_tasks.json`。

#### 2.3.3 动作空间设计

遵循 CoALA 框架[5]的分类方式，系统将动作空间分为外部动作（与浏览器环境交互）和内部动作（与记忆和推理系统交互），共计 21 个动作类型。

**外部动作（16 个）** 覆盖五大浏览器操作类别，如表 1 所示。

**表 1：外部动作空间定义**

| 分类 | 动作名称 | 关键参数 | Playwright 映射 | 说明 |
|:-----|:--------|:--------|:----------------|:-----|
| 元素交互 | CLICK | element_id: int | locator.click() | 通过 SoM 编号定位点击 |
| | MOUSE_CLICK | x: int, y: int | mouse.click(x, y) | Canvas/WebGL/iframe 等盲区的坐标级 fallback |
| | TYPE | element_id, text | locator.fill(text) | 聚焦→清除→逐字输入→确认 |
| | HOVER | element_id: int | locator.hover() | 触发悬停菜单/下拉 |
| 页面操作 | SCROLL | direction: str | mouse.wheel(0, delta) | 向上或向下滚动一个视口高度 |
| | PRESS | key: str | keyboard.press(key) | 键盘单键（与 TYPE 针对元素不同） |
| | WAIT | condition: str | wait_for_load_state() | 语义级等待（load/domcontentloaded/networkidle/commit） |
| 导航 | GOTO | url: str | page.goto(url) | 导航到指定 URL |
| | GO_BACK | — | page.go_back() | 浏览器后退 |
| | GO_FORWARD | — | page.go_forward() | 浏览器前进 |
| 标签页 | NEW_TAB | url: str or None | context.new_page() | 新建标签页，可选指定 URL |
| | CLOSE_TAB | — | page.close() | 关闭当前标签页 |
| | SWITCH_TAB | tab_index: int | pages[i].bring_to_front() | 切换到指定索引标签页 |
| 信息获取 | SCREENSHOT | — | page.screenshot() | 截取当前视口，经 PIL optimize 后转为 base64 data URI |
| | EXTRACT | element_id: int or None | inner_text() | 提取指定元素或页面全文文本 |
| | FIND | text: str | get_by_text(text) | 页内文本匹配搜索，自动滚动到目标 |

**内部动作（5 个）** 服务于推理与记忆操作：THINK 在 ReAct Thought 环节显式记录推理过程到工作记忆；REMEMBER 将关键信息（如"商品价格=2999"）以键值对形式存入工作记忆；RECALL 从工作记忆中按查询字符串检索之前存储的信息；DONE 标记子任务完成并携带摘要；FAIL 标记子任务无法完成并携带失败原因。

所有动作使用统一的 `Action` 数据类承载，字段包含 `action_type`（动作类型枚举）、`element_id`（SoM 元素引用）、`text`（文本参数）、`extra`（扩展参数字典，承载各动作的类型特定参数）和 `confidence`（VLM 生成此动作时的置信度）。

动作空间的设计遵循三项原则：（1）最小充分集——每个动作对应 Playwright 一条或数条原子调用，不做高层抽象封装；（2）VLM 友好——动作类型清晰无歧义，避免过多相近选项增加 VLM 选择负担；（3）可扩展——通过 `extra` 字典承载类型相关参数，新增动作不改变 `Action` 数据类结构。

#### 2.3.4 记忆模块

记忆模块实现三层架构，如表 2 所示。

**表 2：三层记忆架构**

| 记忆层 | 维护者 | 生命周期 | 内容 | 用途 |
|:-------|:------|:--------|:-----|:-----|
| observations | VLM | 单个子任务内 | VLM memory 字段的自然语言拼接 | SubAgent 返回给 Director 的结果载体 |
| working | VLM / Executor | 单个子任务内 | 最近 5 步操作步骤记录 | VLM 在 ReAct 循环中回溯最近操作 |
| knowledge | （预留） | 跨任务 | 结构化的知识条目（分类/键/值/来源） | 跨任务经验积累和复用 |

observations 层是连接 VLM 和 LLM 的桥梁。VLM 在每步输出中包含 `memory` 字段，由 Planner 自动调用 `memory.add_observation()` 追加。当 SubAgent 完成（或失败）时，`memory.get_observations()` 将所有摘要拼接为单一文本串返回给 Director。每个子任务开始前 observations 被清空，确保子任务间的信息隔离。

working 层为 VLM 提供短期上下文。Executor 在每步执行动作后自动向 working 追加条目（格式为 `[ACTION_TYPE] message`）。Planner 在构建 VLM 请求时，从 working 中取最近 5 条记录作为 `recent_actions` 字段，帮助 VLM 理解刚刚完成了什么操作、结果如何。

### 2.4 工程实现与开发规划

#### 2.4.1 技术栈与基础设施

系统基于 Python（3.11 及以上版本）实现，采用 Poetry 作为包管理工具。核心依赖包括：OpenAI 兼容 API 客户端（`openai`）用于 LLM 和 VLM 的统一调用接口；Playwright（1.60 及以上）用于 Chromium 浏览器的自动化驱动；Pillow 用于截图的 PNG optimize 压缩编码；python-dotenv 用于环境变量管理；PyYAML 用于提示词模板的加载与渲染。系统采用标准 `src layout`（PEP 517/518），导入路径为 `classic_web_agent.*`。

配置管理采用三级优先级机制（环境变量 > `config.json` > 代码默认值），通过 `config.py` 中的深度合并器实现。LLM 和 VLM 的 API 密钥、端点地址和模型名称可分别通过 `LLM_*` 和 `VLM_*` 前缀的环境变量配置，支持任何 OpenAI 兼容 API 提供商。

#### 2.4.2 提示词工程

系统使用三类 YAML 格式的结构化提示词模板，由代码在运行时动态渲染填充：

- **`director.yaml`**（LLM）：定义 Director 的角色定位（网页任务规划与协调代理）、SubAgent 能力描述（能做和不能做的操作）、子任务编写规则（粒度/URL/信息指定/依赖处理/动态调整）、输出格式规范（首次 plan 和后续 review 两种 JSON 结构），以及完整的任务计划书书写范例。
- **`reporter.yaml`**（LLM）：定义报告生成的系统指令，支持 Markdown 和 HTML 两种输出格式。报告生成时会将 Director 的完整对话历史作为上下文传入，使 LLM 能看到任务分解和审查的完整过程。
- **`planner.yaml`**（VLM）：定义 VLM 的角色定位、输入格式说明（截图 + 增强 DOM 树文本）、输出 JSON 结构（thinking/verification/memory/goal/actions/confidence 六字段）、21 个动作的详解（分类/参数/风险说明），以及操作规则和约束。

其中，TC-04、TC-08、TC-09、TC-10 需要 LLM 深度参与——LLM 需要利用世界知识拓展研究维度、指定数据源，在子任务执行过程中审查结果并动态调整后续计划，最终生成综合分析报告。

### 3.3 评估指标体系

参考 VisualWebArena[2] 和 WebArena 系列工作的评测设计思路，本实验采用"结果指标 + 过程指标"相结合的三维度加权评分体系。总分计算公式如式（1）所示：

$$Score = I_{complete} \times (S_{task} \times 0.5 + S_{info} \times 0.3 + R_{sub} \times 0.2) \quad \text{(1)}$$

其中各维度的定义和计算方式如下：

**任务完成度** $S_{task}$（权重 0.5）。由独立的 LLM 评估器按照每个测试用例预定义的评分标准（eval_criteria）进行 0-100 分的独立评分。评分标准按测试用例的具体要求将总分拆分为若干子项（例如 TC-01 将 100 分拆分为：是否包含当前温度及温度范围 0-30 分、是否包含降水概率 0-25 分、是否包含风力信息 0-25 分、是否给出户外活动建议 0-20 分）。评估器仅依据最终报告内容进行打分，不参考运行日志。

**信息质量** $S_{info}$（权重 0.3）。由三个子维度加权计算，如式（2）所示：

$$S_{info} = S_{accuracy} \times 0.4 + S_{specificity} \times 0.3 + S_{source} \times 0.3 \quad \text{(2)}$$

其中 $S_{accuracy}$ 为数据准确性（报告中的数字、事实是否准确，0-100 分），$S_{specificity}$ 为信息具体性（是否给出具体数值而非笼统描述，0-100 分），$S_{source}$ 为来源可信度（是否标注了信息来源及其可信程度，0-100 分）。

**子任务完成率** $R_{sub}$（权重 0.2）。$R_{sub} = \frac{n_{completed}}{n_{total}} \times 100$，其中 $n_{completed}$ 为已完成子任务数，$n_{total}$ 为子任务总数。数据直接来源于运行时记录的结构化文件 `sub_tasks.json`，该文件包含每个子任务的 id、goal、status（completed/failed/unknown）和 summary 字段。

**完成系数** $I_{complete}$。为主任务是否完成的二值指示变量（0 或 1），由系统运行状态判定。主任务成功完成（Agent.run 正常返回且 success 为 True）时取 1，否则取 0。该系数确保未完成任务的得分为 0，避免低质量的"假完成"报告获得有意义的得分。

该评估体系的优势在于同时关注"结果做得怎么样"（任务完成度、信息质量）和"过程做得怎么样"（子任务完成率），能够区分不同维度的表现差异，为后续优化提供明确方向。

## 4 实验结果与分析

### 4.1 实验结果汇总

截至本文撰写时，已完成 TC-01（天气查询）和 TC-02（高铁查询）两个测试用例的完整端到端实验。实验结果汇总如表 6 所示。

**表 6：实验结果汇总**

| 编号 | 任务 | 总分 | 任务完成度 | 信息质量 | 子任务完成率 |
|:----:|:-----|:---:|:---------:|:--------:|:-----------:|
| TC-01 | 北京天气查询 | 69.0 | 90.0 | 80.0 | 0/5 (0.0%) |
| TC-02 | 北京至上海高铁查询 | 75.3 | 90.0 | 72.5 | 3/7 (42.9%) |
| 平均 | — | 72.2 | 90.0 | 76.2 | 21.4% |

从汇总数据可以看出两个显著特征：第一，任务完成度（平均 90.0 分）远高于子任务完成率（平均 21.4%），这一反差揭示了系统的双层架构特性——Director/LLM 在最终报告生成阶段具有较强的"兜底"推理补全能力；第二，信息质量（平均 76.2 分）低于任务完成度，说明报告的覆盖范围容易做到，但信息的准确性更难保证。

### 4.2 典型用例详细分析

#### 4.2.1 TC-01：北京天气查询

该用例要求查询当天北京的天气信息，包括温度、降水概率、风力和户外活动建议。Director 为这一任务规划了 5 个子任务，分别访问 5 个国际和国内天气数据源（weather.com、accuweather.com、bbc.com/weather、中国天气网及独立降水数据接口）。

**子任务执行结果**：5 个子任务全部以失败告终，失败原因均为"连续 3 次 VLM 低置信度"（VLM confidence 始终低于 0.9 阈值）。所有子任务均未成功提取到实时天气数据。

**最终报告质量**：尽管所有子任务均失败，但 LLM 在报告生成阶段结合自身世界知识生成了评分高达 90 分的任务完成度报告。信息质量评分为 80 分，其中数据准确性 80 分（温度数据为历史气候经验的推断值而非实时数据）、信息具体性 70 分（降水概率仅给出范围值、风力无具体风速）、来源可信度 90 分（明确标注了中国天气网等来源）。

**分析**：该用例揭示了系统在简单信息提取类任务中的典型表现——VLM 在面对复杂网站交互时频繁失败，但 LLM 的世界知识可以在一定程度上"兜底"。这种兜底机制是一把双刃剑：一方面确保了对用户的体验完整性（不会出现"无法回答"的空白报告），另一方面可能掩盖子任务层面的严重执行问题，降低信息的实时性和准确性。

#### 4.2.2 TC-02：北京至上海高铁查询

该用例要求查询 7 月 1 日北京到上海的高铁信息，涵盖最早和最晚发车时间、二等座票价、全程用时和可选车次。Director 规划了 7 个子任务，设计了多层次访问策略：先尝试 12306 网页端查询（子任务 1-2）、携程备选查询（子任务 3）、12306 JSON API 直接访问（子任务 4-5）、通用票价搜索（子任务 6）、票价字段解码（子任务 7）。

**子任务执行结果**：7 个子任务中 3 个成功、4 个失败。成功的是 12306 JSON API 直接访问（子任务 4，成功提取了 37,594 字符的完整 JSON 响应）、G 字头车次提取（子任务 5，提取了 46 个车次，最早为 G547 06:18，最晚为 G35 19:24）和票价解码（子任务 7）。失败的是所有需要通过页面表单交互的子任务（12306 页面查询、时刻表查询、携程查询）以及通用票价搜索。

**最终报告质量**：任务完成度 90 分，信息质量 72.5 分。信息质量子维度中，数据准确性仅 50 分（车次列表中出现两个 G5 的逻辑矛盾，且票价解析值与实际严重不符——实际高铁二等座票价应在 500 元以上，而系统将编码字段 "M100" 解读为 100 元），信息具体性 95 分（提供的数据详实全面），来源可信度 80 分（标注了 12306 API 来源，但解码方法存在逻辑错误）。

**分析**：TC-02 揭示了系统在两个关键维度上的瓶颈。第一，VLM 对复杂网页表单交互的可靠性严重不足——12306 和携程的搜索表单涉及日期选择、站名输入、验证码等多个步骤，VLM 在每一步都缺乏足够的置信度来继续操作。第二，VLM 在领域特定数据格式的解析上存在能力盲区——12306 JSON 中的票价字段使用特殊的编码格式，VLM 的"猜测式解码"完全失效，产出了与真实值相差甚远的错误数据。

### 4.3 失败模式分析

基于已完成的 12 个子任务执行记录，本文识别出三种主要的失败模式，如表 7 所示。

**表 7：失败模式分类统计**

| 失败模式 | 出现次数 | 占比 | 代表场景 | 根因分析 |
|:--------|:------:|:---:|:--------|:--------|
| VLM 低置信度重试耗尽 | 9 | 75.0% | 天气多源查询全部失败、携程/12306 表单操作 | 复杂表单交互的视觉和语义复杂度超过 VLM 能力边界；planner.yaml 中的动作指导未能充分覆盖表单交互的边界情况 |
| 领域数据编码解析错误 | 1 | 8.3% | 12306 票价字段解码 | VLM 缺乏对特定领域数据编码格式的先验知识；视觉感知无法理解编码语义 |
| 信息源不可访问 | 2 | 16.7% | 12306 页面查询、时刻表查询 | 网站反爬虫机制、动态渲染、登录拦截等超出了 Playwright 基础自动化能力 |

从失败模式分布可以看出，VLM 低置信度重试耗尽是最主要的失败原因，占据了子任务失败的 75%。这种失败模式集中于需要多步骤表单交互的场景（输入搜索关键词后还需选择日期、筛选条件等），说明当前 VLM（mimo-v2.5）在复杂网页交互中的视觉理解和动作规划能力仍有较大提升空间。

### 4.4 讨论

#### 4.4.1 双层架构的有效性验证

实验初步验证了 LLM-VLM 双层分离设计的有效性。LLM Director 的任务规划能力表现突出——在 TC-01 中为简单天气查询规划了 5 个备选国际和国内数据源的冗余设计，在 TC-02 中为高铁查询设计了从网页表单到 JSON API 的多层次访问策略。这种多源、多策略的冗余设计体现了架构的鲁棒性——即便多数子任务失败，仍有备用路径能够获取部分有效数据。

VLM SubAgent 在执行简单、直接的操作（导航到公开 URL、提取 JSON API 响应、解析纯文本数据）时表现可靠；但在面对需要复杂交互的场景时，confidence 阈值的保护机制导致了大量"宁可失败也不冒险执行"的保守决策。

#### 4.4.2 评估框架的有效性验证

三维度评分体系成功区分了不同维度的问题模式。任务完成度维度（均分 90.0）反映了报告的覆盖广度，信息质量维度（均分 76.2）揭示了深层的数据准确性问题，子任务完成率维度（21.4%）量化了 VLM 执行层面的可靠性瓶颈。这种区分度为后续优化提供了清晰的方向指引——当前最紧迫的是提升 VLM 在复杂网页交互中的执行可靠性，而非进一步优化 LLM 的规划能力。

#### 4.4.3 当前局限性

当前实验存在以下局限：首先，实验规模较小（仅完成 10 个测试用例中的 2 个），现有结论的统计显著性有限，需要在剩余 8 个用例上进一步验证；其次，所有测试用例均为信息检索和提取类任务，未覆盖登录认证、文件下载、表单提交等更具挑战性的操作类型；第三，评估中使用的 LLM 评分器与报告生成的 LLM 存在潜在的同源偏差（两者可能共享相似的知识和推理模式），后续应考虑引入人工评估作为校准基准；第四，实验未对不同的 LLM 和 VLM 模型组合进行对比测试，无法判断当前性能瓶颈是源于模型选择还是架构设计。

## 5 总结与展望

### 5.1 工作总结

本文面向真实网页环境中的多步任务自动化问题，设计并实现了一个基于 LLM-VLM 双层协作架构的网页多模态 Agent 系统 ClassicWebAgent，并进行了初步的实验验证。主要工作包括以下几个方面：

**（1）完成了完整的系统架构设计与实现。** 系统以 CoALA 认知架构为组织框架，将智能体能力划分为记忆层、动作空间层和决策循环层。在决策层采用创新的 LLM-VLM 双层协作机制——LLM Director 作为战略级决策者负责任务分解、子任务调度和报告生成，VLM SubAgent 作为战术级执行者在子任务内部完成 ReAct 自治执行闭环。系统代码量控制在合理范围内，实现了轻量化的工程目标。

**（2）实现了基于 CDP 三流融合的感知机制。** 感知模块通过 DOM.getDocument、Accessibility.getFullAXTree 和 DOMSnapshot.captureSnapshot 三个 CDP 协议方法并行采集页面信息，构建增强 DOM 树并以五级条件检测可交互元素。`backendNodeId` 作为贯穿采集、序列化与执行全链路的唯一标识符，避免了额外的编号映射开销。

**（3）定义了完整的动作空间和记忆机制。** 系统定义了 16 个外部动作（覆盖元素交互、页面操作、导航、标签页管理、信息获取五大类）和 5 个内部动作（THINK、REMEMBER、RECALL、DONE、FAIL），并通过 confidence 阈值机制过滤低质量动作。三层记忆架构（observations、working、knowledge）支持子任务内的上下文保持和跨层信息传递。

**（4）建立了系统的实验评估框架。** 设计了覆盖 7 个领域的 10 个端到端测试用例和三维度加权评分体系，编写了自动化评估脚本，初步完成了 2 个用例的实验并获得了可量化的评估数据。实验结果显示系统在任务完成度维度表现良好（平均 90.0 分），但 VLM 子任务层的执行可靠性（21.4% 完成率）仍有较大提升空间。

### 5.2 未来展望

基于当前的实验结果和失败模式分析，后续工作将从以下方向展开：

**（1）强化 VLM 的表单交互能力。** 当前 75% 的子任务失败源于 VLM 在复杂表单场景下的低置信度决策。后续将通过 prompt 工程增加对常见表单模式（日期选择器、下拉菜单、验证码处理）的操作指导和示例，或考虑将高频表单操作封装为更高级的 Skill 原语，减少 VLM 的逐步决策负担。

**（2）引入领域知识解析层。** 针对 TC-02 中暴露的数据编码解析错误，计划建立独立的领域特定知识解析规则库（如 12306 票价字段编码映射表），在 Executor 执行 EXTRACT 后自动触发领域知识的后处理，避免 VLM 的"猜测式解码"。

**（3）扩展实验规模与对比测试。** 完成全部 10 个测试用例的实验，开展不同 LLM/VLM 模型组合的对比测试（如替换为 OpenAI GPT-4V、Claude 等），并引入人工评估作为 LLM 自动评分的校准基准。同时增加登录认证、表单提交等更复杂的操作类测试用例，拓宽评估的覆盖范围。

**（4）完善评估体系中的信息来源可靠性机制。** 在评分体系中增加"信息来源可靠性"惩罚项，通过对比报告中的数值与页面实际提取的数值，区分 LLM 世界知识补全与实时浏览获取的信息，更准确地衡量系统的真实能力。

**（5）推进多 VLM 并行化。** 系统架构已预留并行执行接口（SubAgent 之间无共享状态，各自拥有独立的浏览器标签页和 Memory 实例），后续可通过 Python asyncio 实现互不依赖的子任务在多标签页中同时执行，显著提升信息收集类任务的效率。

**（6）引入 Skill 库与自愈机制。** 参考 browser-harness 的设计理念，建立可复用的预定义操作序列库（如"搜索商品并按价格排序"），在 Director 规划阶段优先匹配 Skill 以短路 VLM 的长链决策。同时引入失败自动恢复机制，当子任务失败时自动尝试降级策略（如切换到备选数据源），减少对 LLM 手动审查调度的依赖。
